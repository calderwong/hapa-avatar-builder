## Title

Hapa Avatar Builder

## One-line description

Turn attributable, Hypercore-backed Card Documents into deterministic collaboration spaces that humans and agents can safely revisit, share, and learn from.

## Track

Developer Tools

## What it does

Hapa Avatar Builder is a local-first visual workbench whose Build Week reference path lives inside its 3D Tarot Draw. A user places ordinary Hapa Cards in a deliberate order. Each Card can receive real per-Card Hypercore custody, and the ordered Formation derives a deterministic private namespace. The Gate visualizes that namespace as a spatial meeting context.

The user can save the Formation as one portable Context Card. That durable Card carries safe scene and formation commitments, but not the secret, full private address, join capability, key, credential, or local path. Restoring it rebuilds the exact Formation while remaining disconnected until a fresh, short-lived Gate Pass is supplied.

With explicit human authority, a Context Card can move through origin staging, Overwind acknowledgement, and a source-only `.hapaCatalog` projection, then return at its pinned revision without producing a competing head. Two isolated local profiles can verify the same Card and meet using Hyperswarm, Noise, and Protomux. Inside the Gate, Context Forge freezes selected evidence and asks a local model to propose a new unminted Card. Three peer-blind Wisdom Cards evaluate the same packet from different rules, preserve disagreement, and return tradeoffs to the human.

Hapa is not the submitted company. It is intended as a shared Card protocol and decentralized platform that many applications and companies can use. Avatar Builder is one participant demonstrating that useful execution can leave portable, attributable learning behind.

## What was built during Codex Build Week

Hapa Avatar Builder, its Card system, and the 3D Tarot Draw existed before the event. During Build Week, the project was extended with the focused Stargate reference path: Card-birth Hypercore custody, ordered deterministic namespaces, portable Context Cards, human-gated Catalog interoperability, expiring Gate Passes and a signed two-profile local P2P proof, consented Comment Cards, Truth Constellation, Context Forge, and peer-blind Wisdom Council.

The conservative auditable slice is 22 commits across 124 files, with 19,136 insertions and 326 deletions between `0b793a9` and `f10c2f8`. A large July 13 release-packaging commit is excluded from that headline because it packaged pre-existing work.

## How Codex and GPT-5.6 were used

Codex worked inside the local Hapa ecosystem and Git history to understand the existing architecture before extending it. In an operator-selected GPT-5.6 Sol session, it helped:

- trace Card, Tarot Draw, Hypercore, P2P, Catalog, and Overwind boundaries;
- implement aligned UI, API, and CLI surfaces;
- build deterministic proof and evidence paths;
- distinguish custody from mint, commerce, ownership, and canon;
- diagnose a recursive-worker incident and turn it into process guards;
- record mistakes, attribution, verification state, and reusable protocol learning;
- reconstruct the official event cutoff and separate pre-existing work from the eligible extension.

GPT-5.6 was the build-time reasoning partner, not a hidden runtime dependency. Runtime Card proposals in the demonstrated Context Forge use local Ollama 0.24.0 with `qwen3.5:27b` and remain unminted pending human approval.

## Current proof boundary

The demonstrated P2P route is an encrypted two-profile local proof, not an internet-scale deployment. Hypercore custody does not imply mint, ownership, commerce eligibility, or canon. Local-model output remains proposed and unminted. Physical-phone participation, live Krea attachment, Memorial Corridor, Passport, and the STG-014 product flow are outside the submission video unless separately proven in a current capture.
